Git & Version Control

Master the essentials of version control using Git

Git is a distributed version control system used for tracking changes in source code. It enables collaboration and version control, making it easy to revert to earlier stages of development.

Core Concepts

  • Repository (repo): A storage location for project files and their change history.
  • Commit: A snapshot of your project at a specific point in time.
  • Branch: A separate line of development, allowing experimentation without affecting the main codebase.
  • Merge: Combining changes from one branch into another.
  • Rebase: Reapply changes from one branch onto another to create a linear history.

Basic Git Commands (Every Developer Must Know)

  • git init - Initialize a new Git repository.
  • git clone <url> - Clone a remote repository.
  • git add <file> - Stage a file (or git add . to stage all changes).
  • git commit -m "message" - Commit changes with a descriptive message.
  • git pull - Fetch and merge changes from a remote repository.
  • git push - Push your local changes to a remote repository.

Basic Git Commands

Branching & Collaboration

  • git branch - List all branches.
  • git checkout -b <branch> - Create and switch to a new branch.
  • git merge <branch> - Merge changes from one branch to another.
  • git rebase <branch> - Reapply commits to create a clean commit history.
  • git stash - Temporarily save uncommitted changes.

Advanced Git Commands & Best Practices

  • git log - View commit history.
  • git reset --hard <commit> - Reset to a specific commit, discarding changes after it.
  • git cherry-pick <commit> - Apply a specific commit from one branch to another.
  • git reflog - View the reference logs of repository updates.
  • git tag - Tag a specific commit for versioning.

Best Practices for Git

  • Commit Often: Keep commits small and focused.
  • Clear Messages: Use descriptive commit messages.
  • Use Branches: Isolate new features in separate branches.
  • Pull Before Push: Pull changes from remote before pushing to avoid conflicts.
  • Merge Regularly: Regularly merge branches to reduce conflicts.

Sample Dockerfile for Practice

# Sample Dockerfile for a Node.js Application
FROM node:14-alpine

# Create app directory
WORKDIR /usr/src/app

# Install app dependencies
COPY package*.json ./
RUN npm install

# Copy application code
COPY . .

# Expose port and define startup command
EXPOSE 8080
CMD [ "node", "server.js" ]

Practice building an image from this Dockerfile by running docker build -t my-node-app . and then running the container with docker run -p 8080:8080 my-node-app.

  web:
    image: node:14-alpine
    volumes:
      - .:/usr/src/app
    ports:
      - "3000:3000"
    command: npm start
  db:
    image: mongo
    ports:
      - "27017:27017"

Use this Compose file to spin up a Node.js web service and a MongoDB database, then try connecting the web app to the database.
